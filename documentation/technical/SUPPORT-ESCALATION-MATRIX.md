## Support Overview

### Support Mission

> *"To provide exceptional, timely support that enables our customers to maximize their SiyaBusa ERP investment and achieve their business objectives."*

### Support Principles

| Principle | Description |
|-----------|-------------|
| **Customer First** | Every interaction prioritizes customer success |
| **Ownership** | Take responsibility until resolution |
| **Communication** | Keep customers informed throughout |
| **Quality** | Resolve correctly the first time |
| **Continuous Improvement** | Learn from every interaction |

### Support Scope

#### What's Included

✅ Product functionality questions  
✅ Configuration and setup assistance  
✅ Bug reports and issue resolution  
✅ Performance troubleshooting  
✅ Integration assistance  
✅ Best practice guidance  
✅ Security incident response  
✅ Data recovery assistance  

#### What's Not Included

❌ Custom development (separate engagement)  
❌ Third-party software support  
❌ Data entry or migration services  
❌ Training (separate service)  
❌ Hardware support  
❌ Network/infrastructure support

## Severity Levels

### Severity Definitions

| Level | Name | Description | Business Impact |
|-------|------|-------------|-----------------|
| **P1** | Critical | System down, data loss risk | Complete business stoppage |
| **P2** | High | Major feature unavailable | Significant impact, workaround difficult |
| **P3** | Medium | Feature impaired | Moderate impact, workaround exists |
| **P4** | Low | Minor issue | Minimal impact |

### Severity Examples

#### P1 - Critical
- System completely unavailable
- Data corruption or loss
- Security breach
- Payment processing failure
- All users unable to log in

#### P2 - High
- Major module not functioning
- Payroll cannot be processed
- SARS submission failing
- Reports not generating
- 50%+ of users affected

#### P3 - Medium
- Single feature not working
- Slow performance
- Export function failing
- Isolated user issues
- Non-critical integrations down

#### P4 - Low
- Cosmetic issues
- Feature requests
- Documentation questions
- Minor inconveniences
- How-to questions

### Severity Assignment Authority

| Initial Severity | Can Be Set By |
|------------------|---------------|
| P4 | Tier 1 |
| P3 | Tier 1 |
| P2 | Tier 2, Customer (verified) |
| P1 | Tier 2, Support Manager, Customer (verified) |

## Escalation Procedures

### Internal Escalation Procedure

#### Step 1: Tier 1 → Tier 2

**Trigger Conditions:**
- Issue not resolved within Tier 1 timeframe
- Complex technical issue beyond Tier 1 scope
- Customer requests escalation
- Severity upgrade required

**Handoff Process:**
1. Document all troubleshooting steps taken
2. Gather relevant logs and screenshots
3. Create detailed escalation note
4. Assign to Tier 2 queue
5. Notify customer of escalation

**Required Information:**
- Ticket number
- Customer details
- Issue description
- Steps to reproduce
- Troubleshooting completed
- Business impact

#### Step 2: Tier 2 → Tier 3

**Trigger Conditions:**
- Code-level bug identified
- Data corruption issue
- Security concern
- Issue not resolved within Tier 2 timeframe

**Handoff Process:**
1. Confirm issue requires engineering
2. Create detailed technical summary
3. Attach all relevant data
4. Create engineering ticket (JIRA)
5. Notify Tier 3 on-call engineer
6. Update customer with engineering ETA

#### Step 3: Management Escalation

**Trigger Conditions:**
- SLA breach imminent or occurred
- Customer dissatisfaction
- Resource constraints
- Cross-team coordination needed

**Process:**
1. Contact Support Manager directly
2. Provide situation summary
3. State needed resources/decisions
4. Document in ticket

## Frequently Asked Questions

### For Customers

**Q: How do I check the status of my ticket?**  
A: Log in to support.siyabusa.com or reply to your ticket email.

**Q: How do I escalate an urgent issue?**  
A: Email escalations@siyabusa.com or call and request a supervisor.

**Q: What information should I include when reporting an issue?**  
A: Steps to reproduce, error messages, screenshots, and business impact.

**Q: Can I request a specific support person?**  
A: Enterprise customers may request specific resources; contact your CSM.

**Q: What if the system is down outside business hours?**  
A: Enterprise customers have 24/7 support. Others should use the emergency line for P1 issues.

### For Support Team

**Q: When should I escalate vs continue troubleshooting?**  
A: Escalate when you've exhausted documented solutions, need code changes, or approaching SLA.

**Q: How do I upgrade/downgrade severity?**  
A: Document justification and update ticket. P1/P2 changes require manager approval.

**Q: What if the customer disagrees with severity assessment?**  
A: Discuss business impact. If unresolved, escalate to Support Manager.

**Q: When should I engage Engineering?**  
A: When issue requires code changes, is a potential bug, or involves data corruption.
